Scope Boundary · 范围边界

清晰的产品化边界

页面上所有架构选择都围绕这个边界展开：Skill 需要独立可用，Agent 负责编排，Tool/MCP/CLI 执行原子动作；真实群发、线上写状态和高风险动作继续由人工确认门禁控制。

Skill 自包含：每个 Skill 包含 SKILL.md、scripts、references、assets 和触发测试。
逻辑下沉：QueryPlan、SQL、分级、通知草稿、send_plan、readiness 均由 Skill scripts 承载。
项目级编排：tools/runners 只负责端到端串联、eval 产物、真实发送门禁和回归入口。
安全门禁：真实群发、拉人入群、线上状态写入仍需人工确认，不由 Skill 默认执行。

Architecture Principles · 架构原则

架构原则不是口号，是约束 Agent行为的规则

架构目标是将业务规则、数据口径、工具权限、验证机制沉淀为可读取、可执行、可审计的结构化约束。

Agent负责编排，不负责猜测

Agent判断入口、选择场景包、调度 Skill、维护状态机和审计链路。它不能绕过场景包直接判断指标口径、Owner（责任人）或升级规则。

意图识别 状态机 调度

Skill负责方法，也要能独立运行

感知、分析、通知、解决是通用能力。每个 Skill 都必须自带操作手册、脚本、参考资料、资产和触发测试；业务事实仍由 `scenario_package_ref` 和 scenario-index 定位。

感知 分析 通知 解决

数据查询必须先治理后执行

感知/分析必须先命中语义层、指标契约或治理数据集，生成查询计划（QueryPlan）后才能执行只读查询，避免字段选择偏移。

语义层优先 查询计划（QueryPlan）

工具层（Tool/MCP/CLI）只做原子动作

查询、推送、写表、回测、导出都在工具层执行；Tool/MCP/CLI必须白名单、限权限、限参数、可审计。

只读优先 白名单 审计

Tool/MCP/CLI的边界规则

Tool（工具）负责线上实时原子动作，例如查指标、查明细、发消息、写事件状态。要求：所有写操作必须带权限策略和人工确认标记。
MCP（工具协议/服务）负责统一工具发现、参数校验、路径白名单、权限拦截和审计记录。要求：知识检索必须校验检索策略（retrieval_policy）。
CLI（命令行工具）负责离线批处理、配置校验、回测、导出、巡检诊断。要求：不上线实时改业务状态，避免和 Tool（工具）职责重叠。
冲突规则同一能力若同时有 Tool 和 CLI，线上执行走 Tool/MCP，离线验证走 CLI。审计链必须统一回写工具调用记录。

Architecture Decision Record · 架构决策记录

四个通用 Skill 的产品化边界与场景流程包引用策略

本节记录能力拆分、场景治理和产品化策略。结论是：稳定动作沉淀为 Skill scripts，变化事实沉淀在 human_review_ops/references/scenarios；单个 Skill 也必须具备独立安装、独立 dry-run 和独立校验能力。

背景

人审运营覆盖效率、质量、成本、体验、安全、合规等模块。模块下会持续新增多个指标、多个分析维度和多种任务类型，不能把某个模块固化成单一场景。

稳定部分感知、分析、通知、解决等动作类型
变化部分指标口径、数据字段、Owner 路由、SLA、通知模板

决策

采用“四个通用 Skill + 根目录场景流程包 + Skill 自包含发布包”的结构。Skill 承载通用能力方法和可执行脚本，根目录场景包承载完整业务事实；发布时由 Skill 内 references/assets/scripts 保证脱离当前 Agent 也可运行。

Skill感知、分析、通知、解决
场景包模块 + 指标/对象 + 任务类型 + 维度规则
发布门禁validate_skill_productization.py --strict + validate_skill_standalone_smoke.py

备选方案

曾考虑按场景拆 Skill、按系统拆 Skill、将知识检索单独成 Skill。最终未采用这些方案，以避免重复建设、流程割裂或能力边界混淆。

每场景一个 Skill短期直观，长期维护成本高
按系统拆 Skill系统边界清晰，但业务闭环被割裂
知识检索独立 Skill易与感知、分析能力耦合

影响

新增运营内容时优先新增或更新根目录场景流程包，而不是复制 Skill。Skill 内快照、assets 和 scripts 必须可由发布清单追踪，保证流程能跑通、口径不漂移、发布可回滚。

收益复用通用能力，降低重复建设
约束场景包必须具备契约、样例、路由、SLA 和发布校验
现状打标率场景已完成四类 Skill 产品化和 AgentBuddy restricted 发布

Data Assets & Contracts · 数据资产与核心契约

数据资产与核心契约

数据契约用于替代非结构化运行时约定，使 Agent、Skill、Tool/MCP/CLI 之间的输入输出可校验、可审计、可沉淀。

统一事件对象

所有正式闭环问题先转成事件对象，作为后续感知、分析、通知和解决的共同输入。

事件 ID（event_id）事件唯一标识
入口来源（source）来源与原文
事件画像（event_profile）场景、指标、范围、等级
证据包（evidence）数据、截图、链接、引用
严重等级（severity）P0-P3（事件等级），P0 最高、P3 最低
流程状态（workflow）当前状态

分析结果对象

分析 Skill输出的结构化结论，支撑人工验收、通知路由和后续回测。

影响评估（impact_assessment）影响范围、规模和等级
候选根因（root_cause）可能原因与证据
SOP 判定（sop_decision）命中规则与处理建议
质量检查（quality_checks）证据与口径检查
只读探索（custom_readonly）不写入业务系统的探索记录

处理结果对象

解决 Skill回收闭环结论，决定是否关闭事件、沉淀案例或进入迭代候选池。

处理动作（actions）处理 / 催办 / 升级
闭环材料（closure）闭环三件套
最终根因（root_cause）复核后的根因
最终结论（final_conclusion）处理结果和业务判断
后续标记（follow_up）迭代标记

查询治理对象

感知/分析阶段必须记录指标映射、查询计划、质量检查和来源脚注，避免模型自由找表导致偏移。

指标契约（metric_contract）唯一口径、公式、过滤条件
查询计划（query_plan）指标、字段、维度、时间窗口
来源层级（source_tier）语义层 / 治理数据集
质量检查（quality_checks）新鲜度 / 完整性
来源脚注（source_footer）口径 / Owner（责任人） / 置信度

工具 / CLI 调用记录

Tool/MCP/CLI的调用轨迹用于排障、审计、权限复核和后续回测，不让工具执行变成黑盒。

调用 ID（tool_call_id）调用唯一标识
调用方（caller）Agent或 Skill
工具标识（tool_name）MCP / Tool / CLI 工具标识
命令名称（command_name）CLI（命令行工具）白名单命令
权限等级（permission_level）只读 / 可写 / 需确认
调用状态（status）成功 / 失败 / 降级

核心资产清单

以下资产是感知、分析和检索治理的前置条件。未完成资产建设的场景包不得进入自动调度范围。

指标契约（metric_contract.md）唯一口径、公式、排除项、维度、时间窗口
数据集说明（dataset_reference.md）推荐字段、混淆字段、禁用字段、血缘和刷新
评估样例（eval_samples）正例、反例、边界样例、跨场景冲突样例

来源脚注样例来源、口径版本、刷新时间、Owner、置信度

Scenario Packages · 场景流程包

一个运营模块下，可以持续沉淀多个场景流程包

这里不要把“模块”等同于“单一场景”。效率、质量、成本等是运营模块；每个模块下会有多个指标、多个分析维度、多个日常/预警/专项任务。场景流程包承载的是某个可执行任务的状态机、SLA（响应/处理时限）、指标契约、Owner（责任人）路由、通知模板和工具策略。

效率模块：多指标运营场景

机审策略有效性只是效率模块下的一个样例方向。同一模块还会包含打标率、自动处置准确率、处置量、时效、人效等指标，并按策略、队列、风险域、时间窗口等维度分析。

打标率 自动处置准确率 处置量 / 时效 策略 队列 / 风险域

质量模块：准确率与事故场景

质量模块不只包含质检准确率，也会包含质检量、误判量、漏判量、底线事故数、事故分层等指标，并按风险域、队列、标签、CQC（质检负责人）等维度拆解。

质检准确率 底线事故数 误判 / 漏判 风险域 队列 / 标签 CQC（质检负责人）

成本模块及其他模块：持续新增

成本模块同样会沉淀多个指标和任务，例如人力成本、队列成本、治理成本、单位处置成本、投入产出比。后续体验、安全、合规等模块也按同样方式新增。

人力成本 单位处置成本 队列成本 治理成本 投入产出比

模块、指标、维度、任务类型的关系

运营模块效率、质量、成本、体验、安全、合规等大类。模块只是管理边界，不等于单一场景。
指标 / 对象例如打标率、自动处置准确率、质检准确率、底线事故数、单位处置成本。一个模块下可以持续新增多个指标。
分析维度例如策略、队列、风险域、标签、CQC（质检负责人）、时间窗口。维度是分析拆解方式，不单独等同于场景。
任务类型日常分析、周期推送、撞线预警、专项治理、复盘追踪。场景包通常由“模块 + 指标/对象 + 任务类型 + 维度规则”组合形成。

场景包最小结构

指标契约（metric_contract.md）唯一指标口径、维度、时间窗口、字段规则
数据集说明（dataset_reference.md）Aeolus（风神查数平台）/ Hive（数仓表）字段选择、混淆字段、禁用字段
感知规则（perception.md）如何识别场景、任务类型和数据就绪
分析规则（analysis.md）查询计划（QueryPlan）、归因、规则命中、来源脚注
样例集（examples.md）正例、反例、边界样例和回测断言

当前产品化状态与安全门禁

通知 Skill 已支持 POC 路由、通知草稿、send_plan、Card 和报表生成；解决 Skill 已支持 manual_tracking。真实群发、open_id 解析、拉人入群和线上写状态仍由人工确认门禁控制。

POC 路由（mach_root_label_name） 发送计划（send_plan）默认阻断群发 人工跟踪（manual_tracking）本地记录

场景包维护责任表

新增运营内容修改：场景清单、状态机、指标契约、数据集说明、样例集。责任：运营负责人、数据负责人、系统负责人共同确认并合入发布。
指标口径变化修改：指标契约、查询计划断言、评估样例、来源脚注版本。责任：数据负责人主责，运营负责人验收业务解释是否正确。
字段或表变化修改：数据集说明、推荐字段、混淆字段、禁止字段、血缘说明。责任：数据负责人主责，系统负责人回测是否仍能选对字段。
通知 / SLA（响应/处理时限）变化修改：通知模板、Owner（责任人）路由、SLA（响应/处理时限）规则、升级条件。责任：运营负责人主责，协作方确认通知范围和升级边界。
只改样例修改：样例集（examples）、期望输出（expected_outputs）、查询计划断言（query_plan_assertions）。责任：运营负责人补案例，系统负责人纳入回测集。

变更发布与回滚流程

1. 草稿场景包进入草稿（draft）状态，只允许人工测试，不进入线上检索。产物：变更原因、影响场景、修改文件清单。
2. 评审运营负责人、数据负责人、系统负责人共同确认口径、路径和权限。产物：评审记录、风险点、是否需要灰度。
3. 回测运行正例、反例、混淆字段、禁用来源、跨场景冲突样例。阻断条件：任一 P0/P1 样例失败，或来源脚注缺失。
4. 灰度只对固定人/群、固定场景或固定时间窗口启用。观察：误召回、错字段、低置信度、人工纠错。
5. 全量 / 回滚全量后保留上一版本；触发错口径、错字段、错通知时回滚。回滚目标：上一版场景包和上一版查询计划断言。

能力模块目录

每个 Skill 长什么样、谁来维护、如何扩展

四个 Skill 不再只是说明文档，而是可独立运行的能力包。每个 Skill 至少包含 `SKILL.md`、`scripts/`、`references/`、`assets/` 和 `test-prompts.json`；根目录场景流程包仍是长期业务事实来源。

产品化 Skill 的目录骨架

每个 Skill 都有自己的能力说明（`SKILL.md`），定义能力边界、输入输出、禁止事项、脚本入口和验证命令；稳定执行逻辑放入 scripts，模板、映射表和 schema 放入 assets。

感知 Skill

负责把入口问题映射成“哪个场景、哪个任务、哪个指标、哪个时间窗口、数据是否就绪”。

维护重点入口分类、实体抽取、场景匹配、数据就绪等级
引用资产场景清单（scenario_manifest）、感知规则（perception）、指标契约（metric_contract）、数据集说明（dataset_reference）
不能做不能直接定口径、不能自由找表、不能跳过场景包

分析 Skill

负责从查询计划（QueryPlan）到可复核结论，覆盖日常分析、周期推送、异常归因和规则命中。

维护重点分析模板、质量检查、归因方法、来源脚注
引用资产指标契约（metric_contract）、数据集说明（dataset_reference）、分析规则（analysis）、样例集（examples）
不能做不能绕过查询计划，不能使用禁止字段或废弃口径

通知 Skill

负责把分析结论转成可审阅通知资产，不只是告警，也包括周报、复盘提醒、人工确认和升级。

维护重点通知模板、等级话术、POC 路由、send_plan、Card 和报表
引用资产通知模板（notification_templates）、责任人路由（owner_routing）、SLA（响应/处理时限）规则（sla）
不能做不能私自扩大通知范围，不能跳过人工确认策略

解决 Skill

负责承接处理状态、人工结论、证据归档、复查和迭代候选，不等同于自动执行业务动作。

维护重点处理状态、闭环三件套、复查规则、迭代候选
引用资产状态机（state_machine）、SLA（响应/处理时限）规则（sla）、样例集（examples）、解决策略（resolution_policy）
不能做不能把未确认结论当闭环，不能自动替代高风险处置

后续如何维护

维护动作按“新增场景、更新口径、更新数据集、更新通知、更新样例”拆开，避免所有人都改同一个 Skill。

新增运营内容新增一个场景流程包，不复制四个 Skill
口径变化只改指标契约，并更新评估样例和版本
字段变化只改数据集说明，补充混淆字段和禁用字段

发布前必须校验

所有场景包变更都要经过结构检查、查询计划断言、样例回测和人工验收，才能进入 Agent 可检索范围。

结构校验必填文件、字段、版本、Owner（责任人）是否完整
回测校验正例命中、反例拒绝、边界样例不误判
发布策略灰度启用、可回滚、记录变更原因

场景文件职责

打标率场景的完整目录不是为了堆文件，而是把“口径、字段、分级、POC 路由、通知、验收”拆到各自负责的文件里。

scenario_manifest声明模块、指标、任务类型、入口关键词、排除条件
metric_contract定义打标率分子、分母、标准样本池 SQL、维度和窗口
dataset_reference指定推荐字段、混淆字段、禁止字段和数据粒度
analysis定义 notice/P2/P1/P0 分级规则、四维粒度和小样本过滤
owner_routing定义 mach_root_label_name 到 POC 姓名级映射和通知规则
eval_samples验证 Agent 是否命中正确场景、指标、字段和输出格式

一次任务如何读取这些文件

1. 感知识别调用 `label_rate_perception.py`，输出 `scenario_key=efficiency-label-rate`、`task_type`、`readiness`。请求真实 SQL、群发或拉人入群时直接阻断。
2. 口径确认读取 metric_contract，确认打标率唯一口径和默认样本池 SQL。未命中指标 ID 或时间窗口时不执行查询。
3. 分析执行调用 `label_rate_analysis.py` 生成 QueryPlan、分级 SQL、source_footer 和标准化结果。外部执行环境只负责真实只读查询和 eval 写入。
4. 通知产物调用 notification scripts 生成 POC 路由、notification_draft、send_plan、Card、报表和飞书表格链接。未传 `sheet_url` 时自动导入 XLSX，默认 `group_send_blocked=true`。
5. 解决记录调用 resolution script 生成 manual_tracking。不写线上状态，不把草稿等同于已发送。
6. 评估回路运行 productization、standalone、分析链路、通知闭环和发布清单 validator。失败则回到 Skill 或场景包修订，不进入发布。

Retrieval & Permission Governance · 检索与工具权限治理

检索策略与工具权限控制

该治理规则贯穿场景包读取和 Tool/MCP 调用：Agent 先完成场景路由，再生成检索策略（retrieval_policy），由 Tool/MCP 层执行路径白名单与权限拦截。

从问题到文档的六步检索链路

1. 识别入口判断是咨询、日常分析、周期推送、预警事件、处理跟进还是复盘。
2. 抽取实体抽取指标、风险域、队列、策略、时间窗口、事件等级和业务对象。
3. 命中场景根据场景清单（scenario_manifest）选择主场景，并记录关联候选场景。
4. 生成策略生成检索策略（retrieval_policy），列出允许路径、拒绝路径和指标 ID。
5. 强制拦截Tool/MCP 层按白名单校验路径；低置信度或多场景冲突转人工确认。
6. 输出脚注在结论里写明来源文件、版本、置信度和未采用的候选原因。

检索策略（retrieval_policy）数据契约

主场景（scenario_key）主场景标识，例如 efficiency-label-rate。用于限定当前任务的主要场景包。
关联场景（related_scenarios）关联候选场景，只允许读取场景清单和冲突排除字段。用于处理跨效率、质量、策略、队列的复杂问题。
允许路径（allowed_paths）允许读取的文件路径，例如当前场景的指标契约和数据集说明。Tool/MCP 层必须按该列表拦截越界读取。
拒绝路径（denied_paths）明确拒绝读取的文件路径、废弃场景、废弃数据集。命中后必须终止检索并输出拒绝原因。
指标 ID（metric_ids）本次任务允许使用的指标 ID。防止同名指标、相近字段和历史口径串用。
置信度阈值（confidence_threshold）场景命中置信度阈值。低于阈值或候选冲突时，进入人工确认，不自动执行查询。

防止被其他文件干扰的机制

场景清单（scenario_manifest）先声明适用范围、入口关键词、排除条件
文档白名单Tool/MCP 层按检索策略（retrieval_policy）强制校验允许路径
指标唯一 ID每个指标绑定 metric_id，避免同名指标串口径
禁止来源数据集说明（dataset_reference）维护禁止来源（forbidden_sources）和混淆字段（confusing_fields）
查询计划断言（query_plan_assertions）自动检查指标、字段、过滤条件是否匹配
来源脚注（source_footer）暴露来源、版本、Owner（责任人）和置信度

A 指标 / A 事件的检索示例

以“近 7 天低打标率策略分级，需要按机审一级标签和 POC 展示”为例，Agent 不会自由找表或直接群发；它应先命中效率模块下的打标率场景，策略 ID、策略名称、送审原因和机审一级标签是受控维度。

输入：近 7 天低打标率策略，按机审一级标签 × 策略 ID × 策略名称 × 送审原因分级
命中：场景标识（scenario_key）= efficiency-label-rate
读取：指标契约（metric_contract.md）确认打标率公式、样本池 SQL、小样本门槛
读取：数据集说明（dataset_reference.md）确认四个推荐维度和禁用字段
读取：分析规则（analysis.md）选择 notice/P2/P1/P0 分级规则
读取：owner_routing.md / POC 映射资产按 mach_root_label_name 生成姓名级 POC
输出：QueryPlan + source_footer + notification_draft + send_plan + manual_tracking

Agent Runtime Governance · Agent 安装与运行约束

Agent 安装、身份与运行约束

Skill说明“怎么做某类能力”，场景包说明“某个业务任务的事实和规则”，而 Agent自身文件说明“我是谁、能安装哪些能力、如何启用、如何拒绝越界任务”。这部分要独立设计，不能散落在 Skill 或场景包里。

Agent自身目录建议

建议把 Agent的身份、能力安装、权限和路由文件独立放在 Agent自身层。身份识别文件（identify.md）可作为别名，但长期建议统一为身份说明（identity.md），避免命名不一致。

human_review_ops/agents/human_review_ops_agent/
调度器总说明（agent.md）目标、职责、边界、默认运行模式
身份说明（identity.md）服务对象、可做/不可做、语气和交互原则
身份识别兼容文件（identify.md）如已有该文件，可映射到身份说明（identity.md）
能力清单（capability_manifest.md）可安装哪些 Skill和版本
安装计划（install_plan.md）启用顺序、依赖、灰度、回滚
路由策略（routing_policy.md）运行模式（run_mode）、任务类型（task_type）、场景包选择规则
权限策略（permission_policy.md）只读、可写、需人工确认、禁止动作
记忆策略（memory_policy.md）哪些经验可沉淀，哪些不能写入长期记忆
评估策略（evaluation_policy.md）安装后如何回测、验收、持续评估

Agent安装能力的流程

安装不是把文件放进去就结束，而是让 Agent明确“启用哪些能力、绑定哪些场景、连接哪些工具、通过哪些校验”。

1. 读取身份读取调度器总说明（agent.md）和身份说明（identity.md），确认 Agent服务范围、职责和禁止事项。结果：形成默认行为边界。
2. 安装能力读取能力清单（capability_manifest.md），启用感知、分析、通知、解决等 Skill。结果：形成可调度能力集合。
3. 绑定场景读取场景清单和场景包，绑定模块、指标、维度、任务类型。结果：形成可检索业务事实来源。
4. 连接工具读取权限策略（permission_policy.md），绑定 Tool/MCP/CLI的白名单和权限等级。结果：形成可审计执行通道。
5. 验收启用读取评估策略（evaluation_policy.md），跑样例回测、反例拒绝、来源脚注检查。结果：通过后才进入可用状态。

Agent文件不承载业务口径

Agent自身文件只定义调度器身份、能力安装和治理边界，不直接写某个指标的公式、字段或负责人。

Agent文件写职责、路由、权限、安装、评估
Skill文件写通用能力方法和输入输出契约
场景包写业务口径、数据集说明、Owner 路由、SLA 和样例

安装后的运行约束

Agent即使安装了某个 Skill，也不能直接自由调用。每次调用都要先看任务类型、检索策略、权限策略和场景包约束。

能力可见由能力清单（capability_manifest）控制
场景可用由路由策略（routing_policy）和场景清单（scenario_manifest）控制
工具可执行由权限策略（permission_policy）和 Tool/MCP 白名单控制

为什么要单独设计这些文件

这样后续新增 Agent、替换 Agent、灰度 Agent或审计 Agent行为时，不需要翻 Skill和场景包里的散落说明。

可迁移换 Agent时可以复用 Skill和场景包
可审计能追溯 Agent为什么能调用某个能力
可回滚能力安装失败时可回滚到上一版安装计划

Agent Orchestration · Agent 编排策略

Agent 编排与局部调度策略

完整闭环是目标形态，但日常运营也需要支持局部能力调用。Agent 根据任务类型（task_type）和运行模式（run_mode）选择最小必要节点，同时加载场景包约束，避免局部任务绕过治理规则。

按任务类型动态选择 Skill的流程

1. 接收任务识别用户是要完整闭环，还是只执行查询分析、责任人定位、通知或处理记录。
2. 判断模式生成运行模式（run_mode）和任务类型（task_type），决定是否进入全流程。
3. 命中场景先用场景清单（scenario_manifest）定位主场景和候选场景。
4. 加载约束按任务类型加载指标契约、数据集说明、责任人路由（owner_routing）、SLA（响应/处理时限）或解决策略。
5. 选择节点只调度必需的 Skill，例如责任人定位只走感知 + 责任人路由。
6. 输出结果返回结果、来源脚注、调用轨迹和未执行节点说明。

调度伪代码

核心原则是“先定场景，再定约束，最后定节点”。即使只运行局部任务，也必须先生成检索策略和约束包。

function dispatch(request):
任务类型（task_type） = classify_task(request) 查询 / 责任人定位 / 通知 / 解决 / 完整闭环
运行模式（run_mode） = decide_run_mode(task_type, request.intent)
场景 = resolve_scenario(request, 场景清单（scenario_manifest）)
检索策略（retrieval_policy） = build_retrieval_policy(场景, 任务类型（task_type）)
约束包（guardrails） = load_required_references(场景, 任务类型（task_type）)
assert validate_guardrails(约束包（guardrails）, 检索策略（retrieval_policy）)
能力节点（skills） = select_skill_nodes(运行模式（run_mode）, 任务类型（task_type）)
result = execute(能力节点（skills）, request, 约束包（guardrails）, 检索策略（retrieval_policy）)
return with_source_footer(result, 约束包（guardrails）, 检索策略（retrieval_policy）)

运行模式（run_mode）和任务类型（task_type）

完整闭环（full_workflow）感知 → 分析 → 通知 → 解决 → 闭环复查
只查询（query_only）感知 + 分析，输出查询计划、结果和来源脚注
责任人定位（owner_lookup_only）感知 + 责任人路由，输出责任人、依据和置信度
只通知（notification_only）基于已有结论生成 Card、POC 路由和 send_plan，真实发送仍需门禁确认
只记录解决（resolution_only）基于已有事件或人工输入记录状态、结论和证据
局部流程（partial_workflow）按用户目标组合多个节点，例如“查询分析 + 责任人定位”

单独查询任务

适合“帮我看某指标上周为什么波动”这类问题，不进入通知和解决闭环。

调度节点感知 Skill + 分析 Skill
必须加载指标契约（metric_contract）、数据集说明（dataset_reference）、分析规则（analysis）
输出查询计划（QueryPlan）、结果、归因、来源脚注（source_footer）

责任人定位任务

责任人定位不是自由搜索通讯录，而是先判断“哪个场景、哪个指标、哪个队列/策略/数据资产”，再执行 Owner（责任人）路由。

调度节点感知 Skill + Owner（责任人）路由规则
必须加载指标契约（metric_contract）、数据集说明（dataset_reference）、Owner（责任人）路由（owner_routing）
为什么要加载口径指标契约确认指标 ID（metric_id）、维度和场景，避免把同名指标路由到错误 Owner（责任人）
为什么要加载数据集说明数据集说明确认队列、策略、数据资产字段，避免用错字段找错人
输出责任人候选、命中依据、置信度、需要人工确认的冲突点

单独解决任务

适合已有人工结论或外部事件状态时，只做处理记录、证据归档和是否复查。

调度节点解决 Skill
必须加载状态机（state_machine）、SLA（响应/处理时限）规则（sla）、解决策略（resolution_policy）
若涉及指标仍需加载指标契约和数据集说明，确认影响口径和复查数据源
输出人工状态、最终结论、证据引用、是否继续观察

局部任务也必须遵守的约束

不能跳过场景识别任何局部任务都要先命中主场景和候选场景。如果置信度不足，进入人工确认，不继续自动执行。
不能跳过检索策略所有知识读取都必须携带检索策略（retrieval_policy）。Tool/MCP 层按允许路径和拒绝路径做强制拦截。
不能跳过指标约束查询分析和责任人定位任务只要涉及指标/事件，就必须加载指标契约。它定义 metric_id、唯一口径、维度、时间窗口和过滤条件。
不能跳过字段约束查询分析和责任人定位任务只要涉及数据资产，就必须加载数据集说明。它定义推荐字段、混淆字段、禁止字段和血缘。
输出必须可追溯局部任务也要输出来源脚注和调用轨迹。没有来源脚注的结果不得进入通知或解决动作。

Runtime Nodes · 运行节点

运行节点、输入输出与审计点

运行节点统一遵循“编排决策 → 能力产出 → 工具执行 → 状态推进 → 审计记录”的交互协议。每个节点都必须声明输入、输出、约束和审计点。

01

感知节点

Agent接收多源输入后，先调用感知 Skill，把自然语言映射成 `scenario_key`、`task_type`、`run_mode`、`metric_ids`、`retrieval_policy` 和 `readiness`。

输入 原始问题、场景提示、指标提示、维度提示、时间窗口
输出 场景标识、任务类型、指标 ID、检索策略、readiness、阻断原因
约束 不执行 SQL、不发通知、不写线上状态；请求真实群发或拉人入群时阻断
审计 记录场景命中、候选场景、拒绝原因和 human_confirmation_required
02

分析节点

分析 Skill 使用 `label_rate_analysis.py` 生成 QueryPlan、打标率 SQL、notice/P2/P1/P0 分级规则、source_footer 和标准化分析结果。

输入 感知输出、指标契约、数据集说明、分析规则、时间窗口和维度
输出 QueryPlan、SQL、readonly_execution、analysis_result、source_footer、provenance
约束 Skill 脚本 dry-run 无副作用；真实只读查询由 runner/Tool 门禁执行
审计 记录查询计划、字段选择、SQL 过滤、小样本门槛和来源脚注
03

通知节点

通知 Skill 使用 notification scripts 将分析结果转成 POC 路由、notification_draft、send_plan、CSV/XLSX 报表和飞书 Card 2.0 草稿。

输入 分级结果、POC 映射、通知模板、表格链接、source_footer
输出 notification_draft、poc_routing_plan、send_plan、Card JSON、报表文件
约束 默认 `group_send_blocked=true`、`requires_confirmation=true`、`sent=false`
审计 记录 POC 依据、卡片哈希、模板版本、发送门禁和人工确认要求
04

解决节点

解决 Skill 使用 manual tracking 脚本承接通知产物，记录人工状态、证据、下一步动作、是否继续观察和闭环三件套。

输入 notification_draft、send_plan、状态机、人工动作和证据引用
输出 manual_tracking、state_transition、closure_check、follow_up
约束 未完成动作、证据、结论三件套不得自动关闭；不写线上状态
审计 记录状态变更、结论来源、证据引用和复查标记
05

工具审计节点

每次 Tool/MCP/CLI调用都记录调用方、工具名或命令名、权限级别、输入输出摘要、耗时、退出码、失败原因和降级动作。

输入 Agent/Skill调用意图、工具参数、CLI（命令行工具）参数、权限策略
输出 调用 ID（tool_call_id）、调用结果、退出码、审计日志、降级建议
约束 写操作必须经过权限策略和人工确认
审计 统一记录调用方、工具名、参数摘要、耗时和失败原因

Workflow State Machine · 流程状态机

从入口到闭环的生命周期

状态机让每个事件都有明确阶段、责任方和退出条件。简单咨询走快路径；需要跟进的事件进入完整闭环。

快路径：不建单但留依据

已接入（INTAKE_RECEIVED） 快路径处理中（FAST_PATH_HANDLING） 快路径已完成（FAST_PATH_CLOSED）
适用 SOP、口径、历史结论查询，或低风险轻量只读查询。

事件闭环：有 Owner（责任人）、有 SLA（响应/处理时限）、有结论

已创建事件（EVENT_CREATED） 感知收集中（PERCEPTION_COLLECTING） 需要补充信息（NEED_MORE_INFO） 分析中（ANALYZING） 等待人工确认（HUMAN_REVIEW_REQUIRED） 已确定责任人（OWNER_ROUTED） 已通知（NOTIFIED） 等待响应（WAITING_RESPONSE） 处理中（IN_PROGRESS） 已升级（ESCALATED） 已提交处理结果（RESOLUTION_SUBMITTED） 闭环审核中（CLOSURE_REVIEW） 已闭环（CLOSED）
人工卡点 分析后、通知前、执行前、闭环前都可按等级和场景配置人工确认。
升级触发 SLA（响应/处理时限）超时、事件等级、重复失败、人工标记都能触发升级。
闭环判定 只有动作完成、证据确认、结论沉淀三件套齐全，事件才能关闭。

Claude Analytics Framework · Claude 数据分析框架

感知/分析如何套用 Claude 官方自助数据分析框架

官方框架的核心结论是：准确性不是 SQL 生成问题，而是上下文、实体映射和验证问题。我们把它落成五层约束。

从业务问题到可验证结论的五层链路

1. 数据基础（Data Foundations）Aeolus（风神查数平台）/ Hive（数仓表）治理数据集、唯一指标口径、字段描述、粒度、血缘和 Owner（责任人）。
2. 事实来源（Sources of Truth）语义层、指标契约、数据集说明、血缘、场景流程包和业务上下文。
3. 知识路由能力（Claude Knowledge Skill（知识路由能力）概念）先识别场景和指标实体，限定允许数据源和禁止数据源；它不是本系统第五个 Skill。
4. 分析流程能力（Claude Runbook Skill（分析流程能力）概念）生成查询计划，执行只读查询、质量检查、归因、规则命中和来源脚注；它落在分析 Skill内。
5. 验证机制（Validation）离线样例、查询计划断言、对抗性复核、线上纠错回收和回测。

防偏移的关键约束

唯一口径公式、排除项、标准过滤固定
泛化规则维度层和时间窗口抽象成规则
字段治理推荐字段、混淆字段、禁止字段
查询计划（QueryPlan）未先写清指标、字段、维度和过滤条件，就不执行查询
来源脚注（source_footer）未说明来源、版本、Owner（责任人）和置信度，就不进入自动通知

必须建设的数据资产（Assets）

这些数据资产（Assets）不是“可选文档”，而是 Agent查询正确性的前置条件。

指标契约（metric_contract.md） 数据集说明（dataset_reference.md） 评估样例（eval_samples） 查询计划断言（query_plan_assertions） 来源脚注样例（source_footer_examples）
