# 感知/分析模块数据查询防偏移规范

## 1. 背景与目标

本文基于 Claude 官方文章《How Anthropic enables self-service data analytics with Claude》的实践框架，约束人审运营 Agent 在感知和分析阶段处理数据、查询指标、解释结果时的行为。

当前 MVP 中，本文是感知和分析模块的核心验收依据。通知和解决模块在测试阶段只做轻量承接，不应稀释对数据查询准确性、指标映射、QueryPlan 和来源脚注的验证。

核心判断：

> 数据分析准确性主要不是 SQL 生成能力问题，而是上下文、实体映射和验证问题。

## 1.1 已确认前提

基于业务补充，当前方案应按以下前提设计：

- 业务核心指标有唯一口径，公式计算、排除项和标准过滤是固定逻辑。
- 维度层和时间窗口虽然更泛化，但可以抽象为规则，例如整体、风险域、三级标签、策略、队列、治理 BP，以及日、周、连续两周、指定时间段。
- Aeolus/Hive 具备字段描述、粒度、血缘等元数据，主要风险不是“无元数据”，而是在字段多、相近含义字段多时，Agent 可能选错字段或误用相似口径。
- 因此治理重点不是重新定义指标，而是把唯一口径、维度规则、时间窗口规则、字段选择规则和禁止字段整理成 Agent 可检索、可执行、可校验的知识输入。
- 可以建设样例和 assets 资产，供 Agent 在感知/分析实践中参考和回测。

在人审运营场景中，模型最容易出现的偏移包括：

- 把用户问题映射到错误指标实体，例如把“打标率”查成别的标签覆盖率。
- 在多个类似数据集之间选错表，或在同一 Aeolus/Hive 数据集中选错相近字段。
- 忽略指标粒度，例如整体、风险域、三级标签、策略、队列、治理 BP 粒度混淆。
- 用过期口径、废弃字段或历史查询逻辑。
- 直接检索历史 SQL 并套用，未确认是否适用于当前场景。
- 结果看起来精确，但没有来源、口径、刷新时间和 Owner。

目标是把感知/分析模块改造成“治理实体优先”的行为模式：

```text
用户问题
  -> 场景识别
  -> 指标/实体映射
  -> 语义层或治理数据集优先
  -> 必要时查 lineage 和场景 reference
  -> 只读查询
  -> 数据质量校验
  -> 分析结论
  -> 来源/口径/置信度/Owner 输出
```

## 2. 三类主要失败模式

| 失败模式 | 在人审运营中的表现 | 治理策略 |
| --- | --- | --- |
| 概念到实体歧义 | “准确率”“处置准确率”“质检准确率”“底线事故”被映射到错误指标、错误数据集或相近字段 | 建设指标语义层、场景指标契约和字段选择规则，强制先查治理指标 |
| 维度/时间窗口泛化 | 指标口径唯一，但整体、风险域、三级标签、策略、队列、日/周/连续两周等粒度选择错误 | 把维度层和时间窗口抽象成规则，进入 QueryPlan 校验 |
| 数据/口径知识过期 | 策略、队列、三级标签、Owner 映射、字段含义变化后，Agent 仍使用旧文档 | 指标、场景流程包和 reference 版本化；CI/评审强制同步 |
| 检索失败 | 正确信息存在于 Aeolus/Hive 元数据或 reference 中，但 Agent 在大量字段、相似字段、历史 SQL 中没有找到正确实体 | 场景流程包限制检索范围，使用 knowledge/runbook 双 Skill 结构 |

## 3. 数据查询信任层级

Agent 在感知/分析阶段必须按以下优先级查找数据实体，不允许直接跳到原始表或历史 SQL。

| 信任层级 | 来源 | 使用规则 |
| --- | --- | --- |
| L1 语义层/指标注册表 | 已治理的指标、维度、口径、粒度、Owner | 默认第一选择；能命中时必须使用 |
| L2 场景指标契约 | 场景流程包中的 `metric_contract.md` 或等价配置 | 用于限定该运营内容允许查询的指标、粒度和过滤条件 |
| L3 治理数据集 | Aeolus/Hive 中已认证的数据集、宽表、事实表 | 仅当语义层未覆盖时使用；必须记录数据集和字段 |
| L4 Lineage/血缘图 | 表血缘、上游模型、下游看板、废弃标记 | 用于判断哪个表是治理来源，哪个表不应使用 |
| L5 结构化 reference | 场景 reference、分析模板、已整理案例 | 用于解释使用方式、过滤条件、常见坑 |
| L6 历史查询语料 | 历史 SQL、看板 SQL、Notebook | 只能作为参考材料，不能作为事实来源直接套用 |
| L7 原始探索 | 原始表、临时表、自由 SQL | 默认禁止；仅专家模式或人工确认后只读探索 |

## 4. 场景指标契约

每项运营内容的场景流程包必须增加指标契约文件：

```text
human_review_ops/references/scenarios/{scenario}/metric_contract.md
```

### 4.1 必备内容

| 内容 | 要求 |
| --- | --- |
| 核心结果指标 | 例如打标率、自动处置准确率、质检准确率、底线事故数 |
| 过程指标 | 例如策略审量、打标量、自动处置量、漏放量、误杀量、策略覆盖、队列处理量 |
| 指标定义 | 分子、分母、过滤条件、排除项、统计周期 |
| 支持粒度 | 整体、风险域、三级标签、策略、队列、治理 BP、审核员、时间周期 |
| 默认时间窗口 | 日、周、连续两周、指定时间段 |
| 允许数据源 | 语义层指标、Aeolus 数据集、Hive 表、看板 |
| 禁止数据源 | 过期表、临时表、非治理口径、历史废弃字段 |
| 数据刷新 | 刷新频率、最新分区、延迟容忍范围 |
| Owner | 指标 Owner、数据 Owner、业务解释 Owner |
| 质量检查 | 新鲜度、完整性、异常值、分母为 0、口径变更 |

### 4.2 示例：机审策略有效性

```markdown
# 机审策略有效性指标契约

## 核心结果指标
- 打标率
- 自动处置准确率

## 默认粒度
- 整体
- 风险域
- 三级标签
- 策略
- 治理 BP

## 默认时间窗口
- 日常低准分析：用户指定时间段
- 周度推送：自然周
- 撞线预警：日级或周级，按规则配置

## 必须优先使用
- 语义层指标：machine_review.label_rate
- 语义层指标：machine_review.auto_disposal_accuracy
- 治理数据集：aeolus_xxx_strategy_effectiveness

## 禁止使用
- 未标记为治理口径的临时策略明细表
- 已废弃的策略效果旧表
- 无 Owner 的历史 SQL

## 标准过滤
- 排除测试策略
- 排除无效队列
- 使用已发布策略版本
```

## 4.3 Aeolus/Hive 数据集 Reference

Aeolus/Hive 已提供字段描述、粒度、血缘等元数据；在人审运营场景中，额外需要补充 LLM 友好的字段选择说明，重点解决相近字段和相近口径的选择偏移。

建议每个核心数据集补充：

```text
human_review_ops/references/scenarios/{scenario}/datasets/{dataset_name}.md
```

模板：

```markdown
# [dataset_name] 数据集说明

## Business Context
说明这个数据集服务哪个运营场景，解决什么分析问题。

## Grain
说明一行数据代表什么，例如策略-天、队列-周、风险域-三级标签-天。

## Recommended Fields
| 业务含义 | 推荐字段 | 使用条件 | 示例 |
| --- | --- | --- | --- |

## Confusing Fields
| 相近字段 | 容易误用原因 | 正确使用方式 |
| --- | --- | --- |

## Forbidden Fields
| 禁用字段 | 禁用原因 | 替代字段 |
| --- | --- | --- |

## Required Filters
列出每次查询必须加的过滤条件。

## Supported Dimensions
列出支持的维度，例如整体、风险域、三级标签、策略、队列、治理 BP。

## Supported Time Windows
列出支持的时间窗口，例如日、周、连续两周、指定时间段。

## Lineage Summary
说明上游来源、下游看板、刷新时间、废弃风险。

## Owner
数据 Owner、业务解释 Owner。
```

## 5. Skill 结构调整：Knowledge + Runbook

Claude 官方实践建议使用 pairwise skills。结合当前架构，感知/分析模块应拆出两类职责：

### 5.1 Knowledge Skill：治理知识路由

职责：

- 根据问题识别业务域、运营内容、指标概念。
- 优先查询语义层/指标注册表。
- 如果语义层不覆盖，则加载对应场景流程包中的 `metric_contract.md`、`analysis.md`、`state_machine.md`。
- 限定可用表、字段、维度和过滤条件。
- 输出“可查询实体清单”，而不是直接生成分析结论。

输出：

```json
{
  "resolved_entities": [
    {
      "concept": "自动处置准确率",
      "entity_type": "metric",
      "entity_id": "machine_review.auto_disposal_accuracy",
      "source_tier": "semantic_layer",
      "grain": ["day", "strategy", "risk_domain"],
      "owner": "string",
      "confidence": 0.98
    }
  ],
  "allowed_sources": [],
  "required_filters": [],
  "forbidden_sources": [],
  "clarification_needed": false
}
```

### 5.2 Runbook Skill：分析执行流程

职责：

- 澄清问题。
- 调用 Knowledge Skill 找到治理实体。
- 生成只读查询计划。
- 执行查询。
- 执行数据质量检查。
- 执行规则命中和归因分析。
- 触发对抗性复核。
- 输出带来源和置信度的结论。

Runbook Skill 不允许绕过 Knowledge Skill 直接查表。

## 6. 感知 Skill 强制行为

感知 Skill 在处理数据问题时必须遵循以下步骤。

| 步骤 | 行为 | 阻断条件 |
| --- | --- | --- |
| 1. 识别场景 | 判断是日常分析、周期推送、撞线预警、事故监控还是临时问答 | 无法识别场景时进入补信息 |
| 2. 识别指标概念 | 抽取指标、粒度、时间范围、对象 | 指标概念不明确时必须澄清 |
| 3. 查语义层 | 优先映射到指标注册表或语义层 | 未查语义层不得直接查原始表 |
| 4. 读取场景指标契约 | 加载 `metric_contract.md` | 场景包缺失时进入人工确认 |
| 5. 限定可用数据源 | 输出 allowed_sources 和 forbidden_sources | 无治理数据源时进入只读探索或人工确认 |
| 6. 检查数据新鲜度 | 校验最新分区、刷新时间、完整性 | 数据过期时不得输出确定结论 |
| 7. 输出证据包 | 记录实体、来源层级、Owner、置信度 | 无证据引用时不得进入通知 |

## 7. 分析 Skill 强制行为

分析 Skill 在查询和解释数据时必须遵循以下步骤。

| 步骤 | 行为 | 阻断条件 |
| --- | --- | --- |
| 1. 接收治理实体 | 只能使用感知输出的 resolved_entities 和 allowed_sources | 禁止重新自由搜索表 |
| 2. 生成查询计划 | 先描述指标、粒度、过滤条件、时间窗口 | 查询计划缺口未补齐时不得执行 |
| 3. 只读工具预检 | 阶段 1 P1 只生成 mock / 只读 `tool_call_record`；接入真实工具后才允许查询治理数据源 | 未经人工确认不得执行真实查询；禁止写入或变更业务状态 |
| 4. 数据质量检查 | 新鲜度、完整性、异常值、分母为 0、口径版本 | 检查失败必须降级置信度 |
| 5. 规则命中 | 对照场景定级/升级规则 | 不允许模型自由解释 P1/P2 |
| 6. 对抗性复核 | 对高风险或领导口径输出进行复核 | 未复核不得标记高置信 |
| 7. 输出来源脚注 | 输出 source tier、数据刷新、Owner、口径版本 | 缺少来源脚注不得闭环 |

## 8. 查询计划契约

任何查询执行前，分析 Skill 必须先生成 `QueryPlan`。


## 9. 查询结果输出规范

所有感知/分析输出必须包含来源脚注。

```text
来源：语义层 / 治理数据集 / 场景 reference / 只读探索
指标口径：metric_id + definition_version
数据时间：统计窗口 + 最新分区时间
粒度：整体 / 风险域 / 三级标签 / 策略 / 队列
Owner：指标 Owner / 数据 Owner
质量检查：新鲜度、完整性、异常检查
置信度：高 / 中 / 低
复核状态：未复核 / 自动复核通过 / 人工复核通过
```

### 9.1 置信度规则

| 置信度 | 条件 |
| --- | --- |
| 高 | 命中语义层或治理数据集，口径明确，数据新鲜，质量检查通过，规则命中明确 |
| 中 | 命中治理数据集但部分维度或过滤条件需说明，或数据存在轻微延迟 |
| 低 | 仅基于只读探索、历史查询参考、字段解释不完整、质量检查部分失败 |

低置信度结论不得自动触发 P1/P2 通知，只能进入人工确认。

## 10. 对抗性复核规范

高风险输出必须经过复核：

- P1/P2 撞线预警。
- 底线事故相关结论。
- 面向负责人 +1 或更高层级的推送。
- 影响范围较大的自动处置准确率或质检准确率异常。
- 需要写入周报、复盘或领导材料的结论。

复核检查项：

| 检查项 | 问题 |
| --- | --- |
| 指标映射 | 是否查了正确指标，而不是相似指标？ |
| 数据源 | 是否使用语义层或治理数据集？ |
| 粒度 | 整体、风险域、策略、队列、标签粒度是否正确？ |
| 时间窗口 | 日、周、连续两周是否按规则处理？ |
| 过滤条件 | 是否应用标准 hygiene filter？ |
| 规则命中 | P1/P2 是否来自配置规则而不是模型主观判断？ |
| Owner 路由 | 负责人是否来自路由数据，而不是模型猜测？ |
| 输出脚注 | 是否带来源、口径、刷新时间、Owner 和置信度？ |

## 11. 离线评估与回归

为避免查询偏移随时间复发，必须建设离线评估集。

### 11.1 样例来源

- 已验证的看板问题。
- 历史撞线预警案例。
- 负责人纠错案例。
- 周度推送案例。
- 质量/效率模块核心指标 Top 问题。
- 长尾业务问题，由运营专家确认标准答案。

### 11.2 样例锚定

| 方式 | 说明 |
| --- | --- |
| 固定快照日期 | 避免实时数据变化导致答案漂移 |
| 固定事实表版本 | 对核心指标使用稳定快照表 |
| 查询计划判分 | 如果数值会变化，则评估是否使用正确指标、表、过滤条件 |
| 规则命中判分 | 评估是否命中正确 P1/P2、升级或通知规则 |

### 11.3 每次评估必须记录

- 场景包版本。
- Skill 版本。
- Tool/MCP/CLI 版本。
- 模型版本。
- Git SHA。
- 语义层命中率。
- 治理数据集命中率。
- 原始探索占比。
- 查询计划通过率。
- 质量检查通过率。
- 对抗复核通过率。
- 人工纠错样例数。

## 12. 线上监控

必须持续监控以下指标：

| 指标 | 含义 |
| --- | --- |
| 语义层命中率 | 感知/分析是否优先使用治理指标 |
| 治理数据集命中率 | 语义层未覆盖时是否使用治理数据集 |
| 原始探索占比 | 原始表或自由查询比例，越高风险越大 |
| 低置信输出占比 | 低置信结论是否被过度使用 |
| 人工纠错率 | 用户指出错表、错口径、缺过滤条件的比例 |
| 静默风险样例 | 事后发现但线上无人纠错的错误 |
| 场景包过期数 | 指标或表变更后 reference 未同步的数量 |

## 13. 维护机制

### 13.1 同仓维护

场景流程包、指标契约、语义层定义和数据模型变更应尽量同仓维护。任何数据模型变更影响核心指标时，必须同步更新：

- `metric_contract.md`
- `analysis.md`
- `owner_routing.md`
- `notification_templates.md`
- 离线评估样例

### 13.2 评审门禁

以下情况不得发布：

- 新增核心指标但没有指标契约。
- 修改指标口径但没有更新场景包。
- 变更数据表但没有 lineage 或 Owner。
- 修改分析 Skill 但离线评估下降。
- 新增 CLI/Tool/MCP 但没有权限和审计策略。

### 13.3 纠错回收

当用户在群聊或评审中指出：

- 查错表。
- 用错口径。
- 少过滤条件。
- 找错 Owner。
- P1/P2 定级错误。

必须进入纠错回收流程：

```text
纠错记录
  -> 定位失败模式：实体歧义 / 数据过期 / 检索失败 / 规则错误
  -> 修改场景包或指标契约
  -> 增加离线评估样例
  -> 回测
  -> 发布
```
