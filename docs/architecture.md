# 人审运营 Agent + Skill + Tool/MCP/CLI 架构设计

## 1. 设计目标

本架构面向人审运营工作流程重构，第一版目标是把“人工经验 + SOP + 部分工具”的工作方式，沉淀为可配置、可追踪、可验收、可迭代的 Agent 编排流程。

架构中的能力分工需要区分三层：

- Agent：负责意图识别、流程编排、状态机、权限卡点、SLA 和升级。
- Skill：负责可复用的业务能力和过程逻辑，例如感知、分析、通知、解决；对已成熟的强绑定样板场景，可收敛为场景级 Skill，并通过路径注册表保留旧四能力路径兼容。
- Tool/MCP/CLI：负责原子工具调用、外部系统连接和命令行能力，例如查指标、读写 Lark Base、发送飞书通知、查询文档、访问数据集、执行批处理或调用内部平台 CLI。

核心使用者是运营值班人员。系统支持自然语言、告警监控、工单、群聊等多源入口，并统一进入简单咨询快路径或事件闭环流程。

第一版优先证明的价值是减少人工步骤，而不是追求全场景覆盖、实时强监控、替代审批或全自动修复。

当前 MVP 范围修正：

- 打标率样板场景已完成感知、分析、通知、解决四段本地闭环，并形成 `efficiency-label-rate-ops` 场景级 Skill。
- 四类通用 Skill 保留为 legacy compatibility path；打标率正式入口默认通过 `skill_path_registry.json` 使用 canonical 场景 Skill，可按需回退 legacy。
- 通知模块已能生成 POC 路由、通知草稿、Card、CSV/XLSX 报表和 `send_plan`；真实群发和在线表格导入必须显式确认或 opt-in。
- 解决模块当前仍采用本地 `manual_tracking.json`，只记录人工处理状态、结论和证据，不写线上状态。
- 下一步重点不再是跑通感知/分析，而是补齐 POC open_id 安全解析、群推送确认链路、状态表 schema、幂等写入和回滚策略。

## 2. 架构原则

| 原则 | 说明 |
| --- | --- |
| 编排优先 | Agent 调度层负责状态机、路由、并行调度、SLA、升级和人工卡点，能力 Skill 不直接决定全局流程。 |
| 能力复用 | 感知、分析、通知、解决能力沉淀为可复用模块；强绑定成熟场景可以封装为场景级 Skill，但必须通过注册表保留旧路径兼容。 |
| 场景独立 | 每项运营内容都有独立状态机、SLA、Owner 路由、通知模板和工具策略，以场景流程包形式维护。 |
| 治理查询优先 | 感知/分析阶段必须优先映射到语义层、指标契约和治理数据集，禁止模型自由找表导致查询偏移。 |
| 工具下沉 | Tool/MCP/CLI 只承载原子能力和外部连接，不直接承载业务流程判断。 |
| 调用可审计 | Agent、Skill、Tool/MCP/CLI 的每次关键调用都应记录输入、输出、权限、耗时和失败原因。 |
| 配置可见 | 运营可维护的流程、规则、Owner、模板、SLA 优先放在 Lark Base。 |
| 只读优先 | 自定义分析第一版只允许只读探索，不直接改变业务状态。 |
| 分级执行 | 执行权限按场景配置，人工卡点按事件等级配置。 |
| 证据闭环 | 事件关闭必须满足动作完成、证据确认、结论沉淀。 |
| 可迭代 | 人工纠错、重复问题、SLA 失败、效果退化都要进入迭代候选池。 |

## 3. 总体架构

```text
多源接入层
  -> Agent 调度与状态机编排层
  -> 可复用能力 Skill 层
  -> Tool/MCP/CLI 连接层
  -> 场景流程包与配置层
  -> 混合存储与评估迭代层
```

### 3.1 多源接入层

输入来源：

- 自然语言提问。
- 告警或监控事件。
- 工单或协作系统。
- 飞书群聊上下文。
- 手工录入。

接入层只做解析和标准化，不直接决定最终动作。所有输入先被识别为 `Consultation` 或 `Event`。

### 3.2 Agent 调度与状态机编排层

职责：

- 意图识别：判断简单咨询、事件闭环、信息不足或非本流程范围。
- 快路径处理：知识问答、轻量只读查询、必要时建议建单。
- 事件创建：把问题标准化为统一事件对象。
- 并行调度：并行调用指标、SOP、Owner、历史案例等能力。
- 工具编排：根据 Skill 需求选择 Tool/MCP/CLI，控制调用权限、参数、运行目录、超时、重试和审计。
- 查询治理：在感知/分析阶段强制执行语义层优先、场景指标契约、查询计划、数据质量检查和来源脚注。
- 状态管理：维护事件状态、SLA 计时、升级、降级、审计。
- 分级卡点：按事件等级决定通知、执行、闭环前是否需要人工确认。
- 权限控制：按场景配置允许动作、禁止动作、确认人和审计字段。

### 3.3 可复用能力 Skill 层

| 能力 | 主要职责 | 输出 |
| --- | --- | --- |
| 感知能力 | 收集事件画像、数据证据、流程上下文、责任配置 | 事件画像、证据包、数据就绪等级、Owner 候选 |
| 分析能力 | 执行影响评估、归因定位、SOP 判定 | 影响结论、候选根因、等级建议、下一步动作 |
| 通知能力 | 仅在用户明确要求通知/找人，或分析结果触发治理、升级、跟进条件时生成通知草稿；后续再接入动态 Owner 路由、催办和升级 | AI Summary 草稿、建议 Owner、发送前检查结果 |
| 解决能力 | 仅在任务进入处置/跟进时记录人工处理状态和结论；后续再接入 SLA 跟进、自动催办、升级和闭环判定 | 人工处理状态、人工结论、证据引用、是否继续跟进 |

Skill 不直接绕过编排层执行高风险动作。所有执行动作都需要读取场景权限配置，并写入审计。

通用 Skill 不直接内置某项运营内容的规则。每次调用必须带上 `scenario_package_ref`，该字段用于定位当前场景包；前期 TRAE 调试态可指向 Skill 内快照，目标治理态应指向根目录 `human_review_ops/references/scenarios/{scenario}`。

当前打标率主线的运行态采用场景级 Skill `efficiency-label-rate-ops` 作为 canonical path。该 Skill 包内包含打标率场景需要的 perception、analysis、notification、resolution 脚本和自包含 references/assets；旧的 `perception`、`analysis`、`notification`、`resolution` 路径保留为 legacy compatibility path。路径选择由 `human_review_ops/configs/skill_path_registry.json` 管理，正式入口通过 `skill_path_resolver.py` 支持 `auto`、`canonical`、`legacy` 三种模式。

Skill 与 Tool/MCP/CLI 的关系：

- Skill 负责“为什么调用、如何解释结果、如何形成业务结论”。
- Tool 负责“执行一个明确的原子动作”，例如查询、发送、写入、下载、检索。
- MCP 负责把外部工具能力以统一协议暴露给 Agent 或 Skill，降低连接成本并统一权限、参数和返回结构。
- CLI 负责执行已有命令行能力，适合批处理、离线评估、配置校验、数据导出、诊断巡检和内部平台命令调用。

### 3.4 Tool/MCP/CLI 连接层

Tool/MCP/CLI 层承载外部系统连接和原子工具能力，不承载完整业务流程。

第一版建议纳入的工具类型：

| 工具类型 | 典型用途 | 调用边界 |
| --- | --- | --- |
| 数据查询 Tool | 查询指标、明细、报表、数据集 | 默认只读；必须记录口径、时间范围、数据源。 |
| Lark Base Tool | 读写流程配置、事件记录、处理结果、验收样例 | 配置写入需要权限和版本；事件关闭前写入三件套。 |
| 通知 Tool | 发送飞书消息、群通知、催办、升级通知 | 发送前检查模板、Owner、SLA 和人工卡点。 |
| 文档/知识 Tool | 读取 SOP、历史文档、复盘结论 | 保留引用来源，不把非结构化结论直接当事实。 |
| 工单/审批 Tool | 创建或查询工单、审批状态 | 不绕过现有审批；高风险动作只做建议或跳转。 |
| 日志/审计 Tool | 记录 Agent、Skill、Tool 调用轨迹 | 关键节点必须记录输入摘要、输出摘要、耗时和错误。 |
| CLI Runner | 执行白名单命令、批处理、回测、导出、校验、巡检 | 只允许预注册命令；必须限制参数、目录、超时和输出脱敏。 |

MCP 的定位：

- 统一工具发现：Agent 可以知道有哪些工具、每个工具需要什么参数。
- 统一调用协议：减少每个 Skill 单独适配外部 API 的成本。
- 统一权限边界：只读、可写、需确认、高风险动作分级控制。
- 统一观测审计：记录工具调用链路、失败原因、重试和降级。

CLI 的定位：

- 复用已有命令行能力：例如内部平台 CLI、数据导出脚本、配置校验脚本、回测脚本。
- 承载离线和批处理任务：例如批量刷新 Owner 映射、生成验收样例、运行回测集。
- 作为 Tool/MCP 的实现形态之一：CLI 可以被包装成 Tool，也可以通过 MCP 暴露为统一工具。
- 必须白名单治理：禁止自由拼接 shell 命令，必须限制命令名、参数 schema、运行目录、超时、输出大小和敏感信息。

### 3.5 场景流程包与配置层

每项运营内容都在根目录 `human_review_ops/references/scenarios/` 下作为完整场景流程包维护，而不是只作为通用 Skill 的补充说明。前期 TRAE 调试时，可把该 Skill 需要的场景文件同步到本 Skill 的 `human_review_ops/skills/{skill}/references/scenarios/` 快照目录，并由 `references/scenario-index.md` 挂载。场景流程包内容：

- SOP 规则。
- 业务场景分类。
- 指标口径。
- 数据 Owner 映射。
- 分析模板。
- 通知模板。
- 场景 SLA。
- 独立状态机。
- 执行权限。
- 人工卡点。
- Tool/MCP/CLI 调用边界。
- 历史案例。

配置字段面向运营可读，后端保留英文映射。配置必须带版本、生效时间、变更原因和评审状态。

推荐包结构：

```text
human_review_ops/references/scenarios/{scenario}/
  analysis.md
  dataset_reference.md
  state_machine.md
  sla.md
  metric_contract.md
  owner_routing.md
  notification_templates.md
  scenario_manifest.md
  examples.md

human_review_ops/skills/{scenario}-ops/
  SKILL.md
  package_manifest.json
  references/
  assets/
  scripts/

human_review_ops/configs/skill_path_registry.json
```

场景级 Skill 是运行态发布包，不替代根目录场景包的业务事实来源。根目录场景包变化后，通过 `build_skill_package.py --target scenario-bundle --write` 同步生成发布包，并通过 `--check-sync` 校验 hash。

### 3.6 混合存储与评估迭代层

一期以 Lark Base 为主：

- 流程定义。
- 规则配置。
- Owner 映射。
- 分析模板。
- 通知模板。
- 场景状态机。
- 场景 SLA。
- 场景指标契约。
- 场景 Tool/MCP/CLI 策略。
- 事件记录。
- 处理结果。
- 历史案例。
- 验收评估。
- 工具调用记录。

后续工程化增强：

- 运行状态进入数据库。
- 审计日志进入日志系统。
- 高频事件结果进入数据库。
- 历史案例可扩展到检索服务。
- Lark Base 保留运营治理视图和配置入口。

## 4. 关键数据流

### 4.1 简单咨询快路径

```text
输入
  -> 意图识别
  -> 判断低风险咨询
  -> 知识问答或轻量只读查询
  -> 输出答案与依据
  -> 命中风险/信息不足/用户要求跟进时转事件
```

快路径不创建正式事件，除非满足以下任一条件：

- 结论置信度不足。
- 查询结果提示潜在业务风险。
- 需要 Owner 处理。
- 用户明确要求跟进。
- 问题涉及 SLA、审批、策略变更或异常处理。

### 4.2 事件闭环主流程

```text
多源输入
  -> 创建事件
  -> 感知收集（通过 Tool/MCP/CLI 查询指标、SOP、Owner、案例）
  -> 数据就绪判断
  -> 标准分析（Skill 解释工具结果并形成结论）
  -> 分级卡点
  -> 数据 Owner 路由
  -> 通知与响应跟踪（通过通知 Tool 执行）
  -> 处理推进（通过工单/通知/记录 Tool 协同）
  -> 结论回收（写入 Base 或结果存储）
  -> 闭环审核
  -> 历史沉淀和迭代判断
```

### 4.3 MVP 轻量流程

当前打标率样板优先使用正式 Skill-first 流程验证完整本地闭环：

```text
用户提问 / 定时任务 / 指标扫描
  -> 感知：识别 efficiency-label-rate、任务类型、run_mode、readiness
  -> 分析：生成 QueryPlan、只读 SQL、分级结果、source_footer、provenance
  -> 通知：生成 POC 路由、notification_draft、send_plan、Card、CSV/XLSX
  -> external_executor：仅在显式确认或 opt-in 时执行真实发送 / 在线表格导入
  -> 解决：生成本地 manual_tracking，记录人工状态、证据和继续观察
```

此流程中，真实群发、在线表格导入、线上状态写入都不由 Skill 默认执行；它们必须由 calling_agent 或 external_executor 完成人工确认、权限检查和审计记录。

## 5. 能力接口

### 5.1 感知能力

输入：

- 原始问题内容。
- 来源上下文。
- 场景配置。
- SOP 规则。
- 数据源和指标口径。
- Owner 映射。
- 可用 Tool/MCP/CLI 清单。
- 场景指标契约和语义层/治理数据集清单。

输出：

- 事件画像。
- 数据证据。
- 流程上下文。
- 责任配置。
- 数据就绪等级：`quick`、`standard`、`deep`、`insufficient`。
- 治理实体映射结果：指标 ID、数据源层级、粒度、Owner、口径版本。

### 5.2 分析能力

输入：

- 统一事件对象。
- 证据包。
- 分析模板。
- SOP 和指标口径。
- 只读查询 Tool 的返回结果。
- 感知 Skill 输出的治理实体和查询计划。

输出：

- 影响评估。
- 候选根因。
- SOP 判定。
- 等级建议。
- 下一步动作建议。
- 质量检查结果。
- 来源脚注：source tier、数据刷新时间、Owner、口径版本、置信度、复核状态。

第一版标准模板：

- 影响评估。
- 归因定位。
- SOP 判定。

### 5.3 通知能力

输入：

- 事件对象。
- 分析结论。
- 数据 Owner 配置。
- 通知模板。
- SLA 和升级策略。
- 通知 Tool/MCP/CLI 调用能力。

输出：

- 接收人。
- 通知渠道。
- 通知文案。
- 必填动作。
- 截止时间。
- 发送状态。
- 升级策略。

主路由是数据 Owner。兜底顺序是场景 Owner、SOP 责任人、备份人、值班机制。

### 5.4 解决能力

输入：

- 事件对象。
- 分析结论。
- 通知状态。
- 场景执行权限。
- SLA 配置。
- 人工确认结果。
- 可用执行 Tool、记录 Tool、工单 Tool 和通知 Tool。

输出：

- 处理动作。
- 催办或升级记录。
- 处理结果。
- 证据确认结果。
- 闭环结论。
- 是否触发迭代。

### 5.5 Tool/MCP/CLI 调用记录

所有关键工具、MCP 和 CLI 调用都应记录为结构化日志，便于排障、审计和回测。

建议字段：

| 字段 | 含义 |
| --- | --- |
| `tool_call_id` | 工具调用唯一标识。 |
| `event_id` | 关联事件 ID，可为空。 |
| `caller` | 调用方，取值为 Agent 或某个 Skill。 |
| `tool_name` | 工具名称、MCP tool 标识或 CLI 命令标识。 |
| `capability_type` | 数据查询、通知、Base 写入、文档读取、工单、审计、CLI 命令等。 |
| `permission_level` | 只读、可写、需确认、高风险。 |
| `input_summary` | 输入参数摘要，避免记录敏感明文。 |
| `output_summary` | 输出结果摘要。 |
| `status` | 成功、失败、超时、降级。 |
| `latency_ms` | 调用耗时。 |
| `fallback_action` | 失败后的降级动作。 |

## 6. 事件分级、SLA 与权限

### 6.1 事件分级

等级采用组合评分：

- 影响范围。
- 业务风险。
- 处理紧急度。
- 结论置信度。

默认等级为 `P0`、`P1`、`P2`、`P3`。等级影响升级强度和人工卡点，不单独决定执行权限。

### 6.2 SLA

SLA 按场景配置，至少包含：

- 响应 SLA。
- 处理 SLA。
- 结论回收 SLA。

### 6.3 权限

执行权限按场景配置，至少包含：

- 允许动作。
- 禁止动作。
- 是否需要人工确认。
- 确认人角色。
- 回滚要求。
- 审计字段。

默认禁止 Agent 自动修改策略、审批、核心业务配置或高风险状态。

## 7. 异常与降级

| 异常 | 降级策略 |
| --- | --- |
| 数据不足 | 进入 `NEED_MORE_INFO`，要求补充字段或人工确认。 |
| Owner 缺失 | 回退到场景 Owner、SOP 责任人、备份人或值班机制。 |
| 分析置信度不足 | 输出低置信度结论，要求人工审核，不进入自动通知或执行。 |
| 通知失败 | 重试、切换渠道、通知备份人并记录失败原因。 |
| SLA 超时 | 触发升级，并记录超时节点、责任对象和下一步动作。 |
| 闭环证据不足 | 退回 `IN_PROGRESS`，要求补充证据或处理结论。 |

## 8. 验收指标

第一版由运营专家人工验收，重点指标：

- 结论一致。
- 路由正确。
- 证据完整。
- 人工步骤减少。

后续回测指标：

- 分类一致率。
- Owner 路由正确率。
- 结论一致率。
- 证据完整率。
- 状态流转正确率。
- SLA 触发正确率。
