# Lark Base 表结构配置草案

## 1. 设计原则

Lark Base 一期作为运营可维护配置和人工验收入口。字段命名优先使用运营可理解的中文名称，后端或自动化脚本可通过英文映射读取。

字段设计原则：

- 字段名去技术化，尽量表达业务含义。
- 每个字段必须有明确描述，避免后续维护人不知道如何填写。
- 规则、模板、SLA、Owner、案例都必须支持版本和启停。
- 事件关闭前必须沉淀动作、证据、结论三件套。

## 2. 通用字段

建议所有配置类表都包含以下通用字段。

| 中文字段 | 英文映射 | 类型 | 必填 | 字段描述 |
| --- | --- | --- | --- | --- |
| 记录ID | record_id | 文本 | 是 | 当前记录的唯一标识，可由系统生成，也可按业务规则维护。 |
| 是否启用 | is_enabled | 单选 | 是 | 控制该条配置是否参与 Agent 路由、分析、通知或执行。 |
| 配置版本 | version | 文本 | 是 | 配置版本号，用于追踪规则、模板、SLA 的变更。 |
| 生效时间 | effective_from | 时间 | 是 | 配置开始生效的时间。 |
| 失效时间 | effective_to | 时间 | 否 | 配置停止生效的时间；为空表示持续生效。 |
| 变更原因 | change_reason | 长文本 | 是 | 说明为什么新增、修改或停用该配置。 |
| 评审状态 | review_status | 单选 | 是 | 可选：草稿、待评审、已发布、已废弃。只有已发布配置可被正式流程使用。 |
| 谁来维护 | maintainer | 人员 | 是 | 负责维护该条配置的人，不等同于事件处理 Owner。 |
| 最近更新人 | updated_by | 人员 | 是 | 最近一次修改该记录的人。 |
| 最近更新时间 | updated_at | 时间 | 是 | 最近一次修改时间。 |

## 3. 业务场景表

用途：定义人审运营中的业务场景、问题分类、默认等级、SLA 和执行权限入口。

| 中文字段 | 英文映射 | 类型 | 必填 | 字段描述 |
| --- | --- | --- | --- | --- |
| 场景名称 | scenario_name | 文本 | 是 | 运营可理解的场景名称，例如“审核质量异常”“申诉复核积压”。 |
| 场景编码 | scenario_code | 文本 | 是 | 给后端或配置引用使用的稳定编码，建议英文小写加下划线。 |
| 业务域 | business_domain | 单选 | 是 | 场景所属业务域，例如审核、质检、申诉、策略、数据。 |
| 问题类型 | problem_types | 多选 | 是 | 当前场景支持的问题类型列表，用于接入后的初步分类。 |
| 入口来源 | source_types | 多选 | 是 | 支持的入口：自然语言、告警、工单、群聊、手工录入。 |
| 触发条件 | trigger_conditions | 长文本 | 是 | 描述什么样的问题应进入该场景，可包含关键词、指标条件、人工判断规则。 |
| 默认事件等级 | default_severity | 单选 | 是 | 未完成组合评分时使用的默认等级，可选 P0、P1、P2、P3。 |
| 允许动作 | allowed_actions | 多选 | 是 | 该场景允许 Agent 编排的动作，例如通知、催办、升级、记录、低风险执行。 |
| 禁止动作 | forbidden_actions | 多选 | 是 | 该场景明确禁止的动作，例如改策略、改审批、直接关闭高风险事件。 |
| 人工确认要求 | confirmation_policy | 长文本 | 是 | 描述哪些节点必须人工确认，例如通知前、执行前、闭环前，或按等级配置。 |
| SLA策略 | sla_policy_id | 关联 | 是 | 关联 SLA 策略配置。 |
| 执行权限策略 | permission_policy_id | 关联 | 是 | 关联执行权限配置。 |
| 默认通知模板 | notification_template_id | 关联 | 否 | 默认使用的通知模板，可被等级或场景细分覆盖。 |
| 适用SOP | sop_refs | 关联 | 否 | 当前场景关联的 SOP、规则或文档。 |
| 备注 | notes | 长文本 | 否 | 其他说明，例如暂不支持的边界或特殊处理方式。 |

## 4. 流程规则表

用途：定义状态机流转、卡点、降级和异常处理。

| 中文字段 | 英文映射 | 类型 | 必填 | 字段描述 |
| --- | --- | --- | --- | --- |
| 规则名称 | rule_name | 文本 | 是 | 规则的业务名称，例如“等待响应超时升级”。 |
| 适用场景 | scenario_code | 关联 | 是 | 该规则适用的业务场景；可配置为通用规则。 |
| 当前状态 | from_state | 单选 | 是 | 状态机中的起始状态。 |
| 触发条件 | trigger_condition | 长文本 | 是 | 触发流转的条件，例如 SLA 超时、人工通过、证据不足。 |
| 下一状态 | to_state | 单选 | 是 | 满足触发条件后的目标状态。 |
| 是否需要人工确认 | requires_human_review | 布尔 | 是 | 该流转是否必须由人确认。 |
| 确认人角色 | reviewer_role | 单选 | 否 | 需要确认时，由运营专家、运营值班、数据 Owner 或业务 Owner 确认。 |
| 降级动作 | fallback_action | 长文本 | 否 | 条件不满足、系统失败或数据缺失时执行的降级处理。 |
| 审计字段要求 | audit_requirements | 长文本 | 是 | 本次流转必须记录哪些字段，例如触发时间、触发原因、执行人。 |
| 优先级 | priority | 数字 | 是 | 多条规则同时命中时的优先级，数字越小优先级越高。 |

## 5. 数据 Owner 表

用途：维护指标、数据集、策略、系统或流程与负责人的映射，是第一版通知路由的主依据。

| 中文字段 | 英文映射 | 类型 | 必填 | 字段描述 |
| --- | --- | --- | --- | --- |
| 数据资产名称 | data_asset_name | 文本 | 是 | 指标、数据集、策略、队列、系统或报表名称。 |
| 数据资产编码 | data_asset_code | 文本 | 是 | 后端可引用的稳定编码。 |
| 资产类型 | asset_type | 单选 | 是 | 可选：指标、数据集、策略、系统、队列、SOP、其他。 |
| 所属业务域 | business_domain | 单选 | 是 | 数据资产所属业务域。 |
| 主负责人 | primary_owner | 人员 | 是 | 默认通知和处理的第一责任人。 |
| 备份负责人 | backup_owner | 人员 | 是 | 主负责人不可达或超时时的兜底负责人。 |
| 升级负责人 | escalation_owner | 人员 | 是 | SLA 失败、高等级事件或重复失败时通知的上一级负责人。 |
| 通知渠道 | notification_channel | 单选 | 是 | 默认通知渠道，例如飞书单聊、飞书群、工单。 |
| 工作时间说明 | working_hours | 文本 | 否 | Owner 可响应的工作时间或值班规则。 |
| 适用问题类型 | applicable_problem_types | 多选 | 否 | 该 Owner 负责的问题类型，留空表示全部适用。 |
| 兜底说明 | fallback_notes | 长文本 | 否 | Owner 缺失、离职、不可达时如何处理。 |

## 6. 分析模板表

用途：维护标准分析模板，第一版优先覆盖影响评估、归因定位、SOP 判定。

| 中文字段 | 英文映射 | 类型 | 必填 | 字段描述 |
| --- | --- | --- | --- | --- |
| 模板名称 | template_name | 文本 | 是 | 运营可理解的分析模板名称。 |
| 模板编码 | template_code | 文本 | 是 | 后端引用的稳定编码，例如 impact_assessment。 |
| 模板类型 | template_type | 单选 | 是 | 可选：影响评估、归因定位、SOP判定、案例对照、只读探索。 |
| 适用场景 | scenario_codes | 多选/关联 | 是 | 该模板适用的业务场景。 |
| 输入字段要求 | required_inputs | 长文本 | 是 | 使用该模板前必须具备的字段和证据。 |
| 输出字段要求 | required_outputs | 长文本 | 是 | 模板必须输出的字段，例如影响范围、候选根因、下一步动作。 |
| 质量检查规则 | quality_checks | 长文本 | 是 | 判断输出是否可用的检查项，例如证据完整、口径一致、Owner 已解析。 |
| 示例输入 | example_input | 长文本 | 否 | 便于运营和开发理解的样例输入。 |
| 示例输出 | example_output | 长文本 | 否 | 便于验收和回测的样例输出。 |
| 只读探索边界 | readonly_boundary | 长文本 | 是 | 描述模板是否允许只读查询，以及允许查询的数据范围。 |

## 7. 通知模板表

用途：维护不同场景、等级、角色的通知口径。

| 中文字段 | 英文映射 | 类型 | 必填 | 字段描述 |
| --- | --- | --- | --- | --- |
| 模板名称 | template_name | 文本 | 是 | 通知模板名称。 |
| 模板编码 | template_code | 文本 | 是 | 后端引用的稳定编码。 |
| 适用场景 | scenario_codes | 多选/关联 | 是 | 该模板适用的场景。 |
| 适用等级 | severity_levels | 多选 | 是 | 适用的事件等级。 |
| 接收角色 | recipient_role | 单选 | 是 | 接收人角色，例如数据 Owner、备份人、升级负责人、运营专家。 |
| 标题模板 | title_template | 文本 | 是 | 通知标题模板，应包含事件等级和场景。 |
| 正文模板 | body_template | 长文本 | 是 | 通知正文模板，应包含摘要、证据、要求动作、截止时间。 |
| 必填证据 | required_evidence | 长文本 | 是 | 发送通知前必须附带的证据，缺失时不能自动发送。 |
| 要求动作 | required_action | 长文本 | 是 | 接收人需要完成的动作，例如确认、处理、回填结论。 |
| 升级文案 | escalation_message | 长文本 | 否 | SLA 超时或高等级事件升级时使用的文案。 |
| 发送前是否确认 | require_review_before_send | 布尔 | 是 | 是否需要运营确认后再发送。 |

## 8. SLA 策略表

用途：按场景定义响应、处理、结论回收时限。

| 中文字段 | 英文映射 | 类型 | 必填 | 字段描述 |
| --- | --- | --- | --- | --- |
| SLA策略名称 | sla_policy_name | 文本 | 是 | 策略名称。 |
| SLA策略编码 | sla_policy_id | 文本 | 是 | 后端引用的稳定编码。 |
| 适用场景 | scenario_codes | 多选/关联 | 是 | 适用的业务场景。 |
| 适用等级 | severity_levels | 多选 | 是 | 适用的事件等级。 |
| 响应时限 | response_deadline | 时长 | 是 | Owner 需要确认收到的时限。 |
| 处理时限 | resolution_deadline | 时长 | 是 | Owner 需要完成处理或给出处理计划的时限。 |
| 结论回收时限 | closure_deadline | 时长 | 是 | 处理后需要回填根因、动作、证据和结论的时限。 |
| 超时动作 | timeout_action | 多选 | 是 | 超时后动作，例如催办、升级、转人工、标记异常。 |
| 升级对象 | escalation_owner_rule | 文本 | 是 | 超时后通知谁，可引用数据 Owner 表。 |
| 是否计入验收 | count_in_evaluation | 布尔 | 是 | 是否纳入 SLA 失败和效果评估。 |

## 9. 事件记录表

用途：记录事件生命周期和当前状态。

| 中文字段 | 英文映射 | 类型 | 必填 | 字段描述 |
| --- | --- | --- | --- | --- |
| 事件ID | event_id | 文本 | 是 | 事件唯一标识。 |
| 来源类型 | source_type | 单选 | 是 | 自然语言、告警、工单、群聊、手工录入。 |
| 来源链接 | source_url | URL | 否 | 原始消息、告警、工单或群聊链接。 |
| 业务场景 | scenario_code | 关联 | 是 | 命中的业务场景。 |
| 问题类型 | problem_type | 文本/单选 | 是 | 命中的问题类型。 |
| 事件等级 | severity_level | 单选 | 是 | P0、P1、P2、P3。 |
| 当前状态 | workflow_state | 单选 | 是 | 当前状态机状态。 |
| 当前责任人 | current_owner | 人员 | 是 | 当前应处理或确认的人。 |
| 数据就绪等级 | readiness_level | 单选 | 是 | quick、standard、deep、insufficient。 |
| 分析结论摘要 | analysis_summary | 长文本 | 否 | 当前最新分析结论。 |
| 证据链接 | evidence_links | URL/附件 | 否 | 关键证据链接或附件。 |
| 响应截止时间 | response_due_at | 时间 | 否 | 响应 SLA 截止时间。 |
| 处理截止时间 | resolution_due_at | 时间 | 否 | 处理 SLA 截止时间。 |
| 结论回收截止时间 | closure_due_at | 时间 | 否 | 结论回收 SLA 截止时间。 |
| 是否已升级 | is_escalated | 布尔 | 是 | 当前事件是否已触发升级。 |
| 异常标记 | abnormal_flags | 多选 | 否 | 信息不足、Owner错误、分析错误、SLA超时、重复问题、处理无效。 |

| 事件ID | event_id | 关联 | 是 | 关联事件。 |

| 证据是否确认 | evidence_confirmed | 布尔 | 是 | 是否已有证据证明问题已处理或影响受控。 |
| 证据链接 | evidence_links | URL/附件 | 是 | 支撑闭环的证据。 |
| 最终结论 | final_conclusion | 长文本 | 是 | 面向复盘和历史沉淀的结论。 |

| 是否触发迭代 | needs_iteration | 布尔 | 是 | 是否进入迭代候选池。 |

## 12. 验收评估表

用途：记录运营专家验收和后续自动化回测结果。

| 中文字段 | 英文映射 | 类型 | 必填 | 字段描述 |
| --- | --- | --- | --- | --- |
| 样例ID | sample_id | 文本 | 是 | 验收或回测样例唯一标识。 |
| 关联事件 | event_id | 关联 | 否 | 如果样例来自真实事件，则关联事件。 |
| 输入摘要 | input_summary | 长文本 | 是 | 样例输入内容摘要。 |
| 专家结论 | expert_conclusion | 长文本 | 是 | 运营专家给出的标准结论。 |
| Agent结论 | agent_conclusion | 长文本 | 是 | Agent 输出的结论。 |
| 结论是否一致 | conclusion_matched | 布尔 | 是 | Agent 结论是否与专家结论一致。 |
| 路由是否正确 | routing_correct | 布尔 | 是 | Owner、通知对象和升级链路是否正确。 |
| 证据是否完整 | evidence_complete | 布尔 | 是 | 关键结论是否有足够证据支撑。 |
| 人工步骤基线 | baseline_manual_steps | 数字 | 是 | 纯人工流程需要的步骤数。 |
| Agent辅助步骤 | agent_assisted_steps | 数字 | 是 | Agent 辅助后仍需人工完成的步骤数。 |
| 验收人 | reviewer | 人员 | 是 | 负责验收的运营专家。 |
| 验收时间 | reviewed_at | 时间 | 是 | 验收完成时间。 |
| 问题标记 | issue_flags | 多选 | 否 | 分类错、路由错、证据不足、结论错、SLA错等。 |
