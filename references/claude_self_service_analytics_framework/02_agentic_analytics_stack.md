# 02 Agentic Analytics Stack

## 1. 总体结构

Claude 官方实践中，提升数据分析 Agent 准确性的核心不是让模型“更会写查询”，而是建设一套分层的数据分析栈。

```text
Data Foundations
  -> Sources of Truth
  -> Skills
  -> Maintenance
  -> Validation
```

对应解决三类错误：

| 层级 | 主要解决的问题 |
| --- | --- |
| Data Foundations | 概念到实体歧义、数据过期 |
| Sources of Truth | 概念到实体歧义、检索失败 |
| Skills | 检索失败、流程不稳定 |
| Maintenance | 数据/口径过期 |
| Validation | 所有剩余错误的发现和闭环 |

## 2. Data Foundations：数据基础

核心目标：让业务概念尽可能映射到唯一治理答案。

关键实践：

- 建设 canonical datasets，即少量、高治理、单一事实来源的数据集。
- 减少近似重复表、重复宽表、重复口径。
- 让治理数据集具备明确 Owner。
- 保持数据模型、语义层、reference 文档、看板定义尽量同仓或同流程维护。
- 把字段描述、粒度、有效范围、血缘、刷新时间、Owner 当成一等产品维护。

在人审运营中的落地：

- 打标率、自动处置准确率、质检准确率、底线事故数必须有唯一治理指标。
- 每个核心指标必须能追踪到 Aeolus 数据集/Hive 表/语义层定义。
- 相似字段必须在 reference 中写明差异和禁用场景。
- 指标口径变更必须同步更新场景流程包。

## 3. Sources of Truth：事实来源

官方建议的信任顺序：

| 优先级 | 来源 | 说明 |
| --- | --- | --- |
| 1 | 语义层 | 已编译的指标和维度定义，命中时必须优先使用 |
| 2 | Lineage/血缘图 | 判断哪个上游模型是治理来源，哪些表已废弃或不应使用 |
| 3 | 结构化 reference | 面向 Agent 检索写的领域文档、表粒度、过滤条件、常见坑 |
| 4 | 业务上下文 | 组织、项目、路标、决策记录、Owner 信息 |
| 5 | 历史查询语料 | 只能作为参考材料，不应直接作为事实来源 |

在人审运营中的落地：

```text
语义层/指标注册表
  -> 场景指标契约 metric_contract.md
  -> Aeolus/Hive 治理数据集
  -> Lineage/字段描述/Owner
  -> 场景 reference
  -> 历史查询样例
```

## 4. Skills：过程知识

官方实践中，Skill 承载的不是事实本身，而是过程知识：

- 应该先查什么。
- 如何处理歧义。
- 如何选择正确数据源。
- 如何执行分析模式。
- 完成分析需要包含哪些校验和输出。

对人审运营的启发：

- 感知 Skill 不应该自由搜索数据，而应先调用 Knowledge Skill 找治理实体。
- 分析 Skill 不应该直接写查询，而应先生成 QueryPlan。
- 场景流程包不是普通说明，而是限制 Agent 检索空间和查询空间的关键资产。

## 5. Maintenance：维护机制

官方实践强调：Skill 和 reference 描述的是每天都可能变化的数据模型，如果不维护，很快会过期。

关键做法：

- Skill/reference 与数据模型变更同仓或同流程维护。
- 数据模型 PR 如果影响指标口径，必须同步更新 Skill/reference。
- 定期清理过期脚手架和过期文档。
- 同一套 Skill 在 IDE、聊天、看板工具、Agent 会话中保持一致。

在人审运营中的落地：

- 策略、队列、三级标签、Owner 映射变化时，同步更新场景包。
- Aeolus/Hive 数据集字段变化时，同步更新 `metric_contract.md`。
- 通知模板、定级规则、升级规则变化时，同步更新场景包和回测样例。

## 6. Validation：验证机制

验证用来持续发现错误是否仍然存在。

关键机制：

- 离线评估。
- 消融实验。
- 线上来源脚注。
- 数据质量检查。
- 被动监控。
- 主动纠错回收。

在人审运营中，最小可行验证包括：

- 每个场景至少 3-5 条验收样例。
- 每条样例固定指标、时间窗口、粒度和期望结论。
- 每次修改场景包后运行回测。
- 用户纠错必须反哺场景包和评估集。
