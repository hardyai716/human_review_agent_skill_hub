# 04 验证与评估框架

## 1. 为什么需要验证

Claude 官方实践强调：如果没有评估体系，很难知道数据分析 Agent 是否真的准确。

在人审运营场景中，感知/分析模块必须证明：

- 是否识别了正确运营内容。
- 是否映射到正确指标。
- 是否选对数据集和字段。
- 是否使用正确粒度和过滤条件。
- 是否命中正确定级/升级规则。
- 是否输出可追溯来源。

## 2. 离线评估

离线评估是固定问题和标准答案的集合。

### 2.1 样例来源

| 来源 | 用途 |
| --- | --- |
| 已验证看板问题 | 覆盖最常见分析问题 |
| 日常分析案例 | 覆盖自然语言分析请求 |
| 周期推送案例 | 覆盖周度推送和摘要生成 |
| 撞线预警案例 | 覆盖 P1/P2 定级和规则命中 |
| 人工纠错案例 | 覆盖历史失败模式 |
| 长尾业务问题 | 覆盖非标准问法 |

### 2.2 样例锚定

为避免数据实时变化导致评估漂移，样例应至少采用一种锚定方式：

- 固定快照日期。
- 固定事实表版本。
- 固定看板结果。
- 判 QueryPlan，而不是判实时数值。
- 判规则命中，而不是判自然语言解释。

### 2.3 离线评估字段

```json
{
  "sample_id": "string",
  "scenario": "machine_review_effectiveness",
  "task_type": "daily_analysis | weekly_push | threshold_alert",
  "question": "string",
  "expected_metrics": [],
  "expected_dimensions": [],
  "expected_time_window": {},
  "expected_filters": [],
  "expected_sources": [],
  "forbidden_sources": [],
  "expected_rule_hit": "P1 | P2 | none",
  "expected_output_shape": "summary | detail | root_cause_report | ai_card"
}
```

## 3. 评估指标

| 指标 | 含义 |
| --- | --- |
| 场景识别准确率 | 是否识别到正确运营内容 |
| 任务类型识别准确率 | 是否识别日常分析、周期推送、撞线预警 |
| 指标映射准确率 | 是否映射到正确指标 |
| 数据源命中率 | 是否使用语义层或治理数据集 |
| 禁止数据源误用率 | 是否误用了禁用表或字段 |
| QueryPlan 通过率 | 查询计划是否符合指标契约 |
| 规则命中准确率 | P1/P2/不触发判断是否正确 |
| 来源脚注完整率 | 是否输出来源、口径、刷新、Owner、置信度 |
| 人工纠错率 | 用户指出查错或解释错的比例 |

## 4. 消融实验

官方实践建议对关键结构做消融，避免凭感觉争论。

在人审运营中可测试：

| 消融项 | 比较目标 |
| --- | --- |
| 有无 `metric_contract.md` | 指标映射准确率是否提升 |
| 有无语义层优先 | 是否减少错表错字段 |
| 有无历史 SQL 检索 | 是否真的提升准确率，或只是引入噪音 |
| 有无 QueryPlan | 查询错误是否下降 |
| 有无对抗性复核 | P1/P2 和底线事故结论是否更稳 |
| 有无来源脚注 | 人工审核效率是否提升 |

要求：

- 固定同一批离线样例。
- 每次只改一个组件。
- 记录通过率变化。
- 记录无效或负向实验，避免重复踩坑。

## 5. 线上验证

线上验证用于监控真实运行质量。

### 5.1 来源脚注

每个分析输出必须带来源脚注：

```text
来源：语义层 / 治理数据集 / 场景 reference / 只读探索
指标口径：metric_id + definition_version
数据时间：统计窗口 + 最新分区时间
粒度：整体 / 风险域 / 三级标签 / 策略 / 队列
Owner：指标 Owner / 数据 Owner
质量检查：新鲜度、完整性、异常检查
置信度：高 / 中 / 低
复核状态：未复核 / 自动复核通过 / 人工复核通过
```

### 5.2 线上监控指标

| 指标 | 说明 |
| --- | --- |
| 语义层命中率 | 越高越好 |
| 治理数据集命中率 | 语义层未覆盖时的兜底质量 |
| 原始探索占比 | 越高风险越大 |
| 低置信输出占比 | 用于判断是否过度自动化 |
| 人工纠错率 | 用户指出错表、错口径、缺过滤的比例 |
| 静默错误样例 | 事后发现但线上无人纠错的错误 |

## 6. 主动纠错回收

纠错来源：

- 用户在群聊指出“查错表”。
- 用户指出“口径不对”。
- 用户指出“少了过滤条件”。
- 用户指出“P1/P2 定级不对”。
- 用户指出“Owner 找错”。

纠错流程：

```text
纠错记录
  -> 归因失败模式
  -> 修改 metric_contract / analysis / owner_routing / rule
  -> 增加离线评估样例
  -> 回测
  -> 发布
```

## 7. 发布门禁

以下情况不建议发布：

- 核心指标没有指标契约。
- QueryPlan 通过率不足。
- 禁止数据源误用仍存在。
- P1/P2 样例无法稳定命中。
- 来源脚注不完整。
- 修改 Skill 后离线评估下降。
- 高风险结论没有复核机制。
