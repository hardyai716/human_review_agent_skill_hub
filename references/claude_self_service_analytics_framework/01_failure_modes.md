# 01 数据分析 Agent 的主要失败模式

## 1. 官方框架中的三类失败

Claude 官方实践将自助数据分析 Agent 的主要错误归为三类。

| 失败模式 | 含义 | 在人审运营中的表现 |
| --- | --- | --- |
| 概念到实体歧义 | 用户说的是业务概念，但数据模型里存在多个相似表、字段、指标，Agent 选错实体 | “打标率”“自动处置准确率”“质检准确率”“底线事故数”被映射到错误字段或错误数据集 |
| 数据/口径过期 | 数据源、字段、指标定义、业务规则持续变化，Agent 使用了旧知识 | 策略、队列、三级标签、Owner 映射、指标过滤条件已经变化，但场景包未同步 |
| 检索失败 | 正确信息存在，但数据空间太大，Agent 没有找到或没有正确使用 | Aeolus/Hive 表字段很多且含义接近，Agent 找到了相似字段但不是治理口径字段 |

## 2. 为什么这不是单纯 SQL 问题

如果 Agent 已经知道：

- 应使用哪个治理指标。
- 应使用哪个数据集或 Hive 表。
- 应使用哪些维度和过滤条件。
- 应使用哪个时间窗口。
- 哪些表和字段禁止使用。

那么 SQL 或查询执行通常不是最难的问题。

真正困难的是：

```text
用户问题
  -> 业务概念
  -> 治理指标
  -> 正确数据实体
  -> 正确粒度
  -> 正确过滤条件
  -> 可验证结果
```

## 3. 对人审运营的约束

在人审运营项目中，必须把以下内容前置为显式资产：

- 指标契约：`metric_contract.md`
- 数据源白名单：允许使用的 Aeolus 数据集、Hive 表、语义层指标
- 数据源黑名单：禁止使用的临时表、废弃表、非治理口径字段
- 字段解释：字段含义、粒度、可用范围、常见误用
- 标准过滤：测试策略、无效队列、无效标签、过期策略版本等排除项
- 版本信息：指标口径版本、场景包版本、数据刷新时间

## 4. 防偏移原则

1. 不允许 Agent 直接从原始表或历史 SQL 开始。
2. 不允许 Agent 自行创造指标定义。
3. 不允许 Agent 在多个相似字段中凭语义猜测。
4. 不允许没有 QueryPlan 就执行查询。
5. 不允许没有来源脚注就输出分析结论。
6. 不允许低置信结论自动触发通知或升级。
